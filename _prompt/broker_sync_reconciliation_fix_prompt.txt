You are working inside an existing MERN trading bot codebase.

Your task is to implement a focused reconciliation fix for stale local broker-sync open trades.

OBJECTIVE

When the broker no longer reports a position, any matching local open `broker_sync` trade must be reconciled automatically so the Journal and summary cards stop showing phantom open positions.

CURRENT BUG

The current behavior is partially correct:

- When Alpaca/broker reports an open position, the app creates or maintains a local open trade with:
  - `strategyName: "broker_sync"` and/or
  - `source: "broker_sync"`

However, when that position is later closed directly in Alpaca, the local broker-sync trade remains open in the app.

This causes:
- Dashboard broker positions to become empty
- But Journal summary still showing `OPEN 1`
- And Journal table still showing a stale `broker_sync` row as `open`

This is a reconciliation bug.

REQUIRED OUTCOME

During broker reconciliation/sync:

- if broker still has the symbol open -> keep or update local broker-sync open trade
- if broker no longer has the symbol open -> automatically reconcile the local broker-sync open trade

The stale local `broker_sync` open trade must no longer remain open once the broker is flat.

SCOPE

Make a focused backend fix only unless a tiny frontend adjustment is absolutely necessary.
Do not redesign UI.
Do not change strategy logic.
Do not change autopilot entry rules.
Do not change deployment config.

Search the repo for the broker sync / reconciliation flow and update the real files involved.

REQUIREMENTS

1. RECONCILE ORPHANED BROKER_SYNC OPEN TRADES
Find the code path that syncs broker positions into local journal/open trade state.

Add logic so that after broker positions are fetched, local open trades are checked.

For each local open trade that is a broker-sync trade:
- if its symbol is NOT present in current broker open positions
- reconcile it automatically

Reconciliation can be done by either:
- marking it closed and moving it to closed history, or
- updating it to a closed/reconciled status using the repo’s existing trade lifecycle conventions

Use the existing lifecycle pattern in the project rather than inventing a second closure system.

2. ONLY APPLY THIS TO BROKER_SYNC-TYPE TRADES
Do not automatically close real strategy-managed open trades from this rule.

This reconciliation behavior must be limited to local trades that clearly represent imported broker positions, such as:
- `strategyName === "broker_sync"`
- or equivalent `strategy/source` field used in the repo

Use the actual schema in the codebase.

3. PRESERVE AUDITABILITY
When reconciling a stale broker-sync open trade, record a meaningful exit/close reason such as:

- `broker_no_longer_open`

Use one concise, production-safe reason string consistently.

If the repo supports closed timestamps and exit metadata, populate them.

4. AVOID DUPLICATE CLOSED RECORDS
The reconciliation must be idempotent.

If the sync runs repeatedly after the broker position is gone, it must not keep creating duplicate closed records for the same stale broker-sync trade.

7. DO NOT JUST HIDE IT IN THE UI

The source-of-truth local trade state must be reconciled correctly in backend storage/state.

8. HANDLE EMPTY BROKER POSITION LIST
Important edge case:
- broker returns zero open positions
- local storage still has one or more open broker-sync trades

All such broker-sync trades should be reconciled cleanly.
