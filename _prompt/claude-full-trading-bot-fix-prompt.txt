1. BROKER-SYNC RISK MANAGEMENT (HIGHEST PRIORITY)
==================================================

Problem

A live broker-sync position can currently appear with:
- stop = null / blank
- target = null / blank
- risk = 0
- still counted as a live open position

This is unsafe because the system may carry live exposure with no defined risk state.

Required outcome

Every live open position must have an explicit management state:
- managed
- derived
- unmanaged

And broker-sync positions must no longer silently appear as zero-risk positions unless they are truly unmanaged and explicitly labeled that way.

Implementation requirements

Backend:

Inspect and update:
- src/journal/tradeJournal.js
- src/positions/exitEngine.js
- src/server/routes/dashboard.js
- any relevant config/env module

Implement broker-sync enrichment logic so that when a broker-sync position is created, merged, or synced, the system attempts to populate risk state.

Add or derive:
- stopLoss
- takeProfit
- riskPerUnit
- riskAmount
- managementStatus: "managed" | "derived" | "unmanaged"
- riskSource: "journal" | "derived" | "none"
- exitCoverage: "full" | "partial" | "none"
- origin: "strategy" | "broker_sync"

Fallback order for broker-sync risk:
1. existing journal stop/target if available
2. ATR-derived stop if ATR/metrics are available
3. fixed-percent stop from env if ATR unavailable
4. if still impossible:
   - mark position unmanaged
   - do NOT report risk as zero unless the UI/API clearly identifies it as unmanaged

Add env support if needed:
- BROKER_SYNC_ENABLE_DERIVED_RISK=true
- BROKER_SYNC_STOP_PCT=0.02
- BROKER_SYNC_TARGET_R_MULTIPLE=2

Behavioral requirements:
- derived broker-sync stops should be usable by the exit engine if enabled
- unmanaged broker-sync positions should not look safe or managed
- log or persist an explicit event when an unmanaged broker-sync position is detected

Dashboard/API requirements:
Expose the enriched fields through dashboard position payloads.

Frontend:
Update dashboard open positions UI so that:
- Broker Sync badge remains
- if derived risk is being used, show subtle label like "Derived"
- if unmanaged, show visible warning badge like "Unmanaged"
- never make an unmanaged broker-sync position look like a valid fully managed strategy trade

3. STRATEGY QUALITY / OVER-REJECTION FIX
==================================================

Problem

The strategy is rejecting far too many candidates and producing weak live performance:
- rejection count is extremely high
- approval count is very low
- expectancy is negative
- win rate is poor

The system is now structurally correct enough to reveal that the strategy logic needs improvement.

Required outcome

Refactor the breakout strategy so it is less brittle and more tunable, while preserving safety and risk checks.

Implementation requirements

Inspect and update:
- src/strategies/breakoutStrategy.js
- src/autopilot.js
- any ranking/scoring helpers already used by the strategy

Required changes

A. Add breakout tolerance / near-breakout classification
Current logic appears too binary. Implement configurable breakout thresholds such as:
- BREAKOUT_CONFIRMATION_PCT
- BREAKOUT_NEAR_MISS_PCT

Classify setups as:
- confirmed_breakout
- near_breakout
- no_breakout

Use near-breakout to improve ranking/analytics even if not approved.

B. Move signal quality toward score-based gating
Keep hard rejects only for:
- insufficient market data
- invalid stop/risk math
- impossible sizing
- hard risk guard failures
- duplicate exposure / max positions / daily loss guard

For signal-quality conditions:
- compute a score
- approve only if score >= threshold
- otherwise reject with score-driven reasons

C. Make volume / ATR logic less brittle
Instead of collapsing all failures into broad reasons, classify them more explicitly:
- weak_volume
- missing_volume
- atr_too_low
- overextended_breakout
- weak_risk_reward

D. Support session-aware thresholding if easy to implement cleanly
Examples:
- MIN_SETUP_SCORE_TOKYO
- MIN_SETUP_SCORE_LONDON
- MIN_SETUP_SCORE_NEW_YORK

Do not overcomplicate if the codebase would suffer, but structure the logic so session-specific thresholds can exist cleanly.

E. Preserve safety
Do not make the strategy reckless.
The fix is not "approve more at any cost".
The goal is:
- better signal discrimination
- fewer useless hard rejects
- better diagnostics
- tunable thresholds
